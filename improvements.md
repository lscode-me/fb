## 2. Часть I. Файлы как элемент файловой системы
- [x] **Таблица лимитов ФС**: Сравнение макс. размера файла и тома для ext4, XFS, NTFS, APFS, exFAT, FAT32.
- [x] **File Locking**: Добавить детали про механизмы блокировок (advisory vs mandatory, `flock`, `fcntl`). Почему базы данных не любят NFS?
- [x] **Бинарные исполняемые файлы**: Краткий обзор структур ELF (Linux) и Mach-O (macOS) в контексте «файл как контейнер кода/данных».
- [x] **Extended Attributes (xattr)**: Расширить примеры использования (например, карантин в macOS или метаданные в SELinux).
- [x] **Memory-Mapped Files (mmap)**: Добавить в главу о дескрипторах или файловых операциях раздел о mmap — проецирование файла в память, когда это эффективнее read/write, подводные камни (coherence, truncation).
- [x] **Filesystem Events / Watches**: inotify (Linux), FSEvents (macOS), ReadDirectoryChangesW (Windows) — как программы узнают об изменениях файлов в реальном времени. Практика: наблюдение за директорией на Python (`watchdog`).
- [x] **Атомарность файловых операций**: Паттерн «write to temp + rename» для безопасной записи. Почему `rename()` атомарен в пределах одной ФС, а дозапись — нет. fsync и его важность.
- [x] **Временные файлы**: `tempfile` (Python), `mktemp` (shell), `/tmp` vs `/var/tmp`. Автоочистка, безопасность (race condition при предсказуемых именах).

## 3. Часть II. Файлы как контейнеры данных
- [x] **Визуализация Endianness**: Схема размещения байт в памяти для Little и Big Endian. Примеры форматов, где это критично.
- [x] **Zero-copy**: Объяснение концепции передачи данных без копирования в памяти (sendfile, splice) — почему это делает веб-серверы быстрыми.
- [x] **SIMD и кодировки**: Упомянуть, как современные процессоры используют векторные инструкции для ускорения валидации UTF-8.
- [x] **BOM в реальном мире**: Практические проблемы с BOM (Byte Order Mark) — конфликты с шебангом, JSON-парсерами, Git diff. Как обнаружить и удалить BOM.

## 4. Часть III. Форматы файлов
- [x] **Бинарные альтернативы JSON**: Краткий обзор MessagePack, CBOR, BSON. Когда стоит переходить с JSON на бинарный формат.
- [x] **SQLite — файл как база данных**: Глубокий разбор формата файла SQLite, ACID, страницы, журнал (WAL). Почему это лучший формат для хранения структурированных данных в одном файле.
- [x] **Parquet Internals**: Описать Predicate Pushdown и Row Groups (почему чтение 1 колонки из 100 работает мгновенно).
- [x] **Schema Registry**: Упомянуть в контексте Avro и Kafka — управление жизненным циклом схем вне самого файла.
- [x] **Таблица-навигатор «Какой формат выбрать»**: Decision matrix: человекочитаемость, размер, скорость парсинга, поддержка схемы, streaming-совместимость. Помогает читателю не теряться в 10+ форматах.
- [x] **JSON Schema и валидация**: Как описывать и проверять структуру JSON-документов. Аналогии с XSD для XML.
- [x] **Формат SQLite WAL подробнее**: Добавить схему страниц и журнала, показать `sqlite3_analyzer` для исследования реального файла.

## 5. Часть IV. Работа с файлами и потоками
- [x] **Python: Durable mappings**: Модули `dbm` и `shelve`. Когда нужно простое хранилище «ключ-значение» без полноценной инфраструктуры БД.
- [x] **DuckDB + S3**: Добавить пример запроса к удаленному Parquet-файлу по HTTP/S3 без полного скачивания (HTTP Range Requests).
- [x] **Обновление списка инструментов**: Добавить `uv` (Python), `dust` (анализ места), `tokei` (статистика кода), `hexyl` (hex-просмотрщик).
- [x] **io_uring**: Упомянуть как современную альтернативу стандартному асинхронному I/O в Linux.
- [x] **Бенчмарки форматов**: Практическое сравнение скорости чтения/записи + размера для CSV vs JSON vs Parquet vs SQLite на одном датасете. Код на Python + DuckDB, воспроизводимый читателем.
- [x] **Pipe и FIFO в деталях**: Именованные каналы (mkfifo), размер буфера pipe, deadlock при полном буфере. Практика: обмен данными между процессами через FIFO.

## 6. Часть V. Инфраструктура хранения
- [x] **Глубокий разбор APFS**: Контейнеры (Containers), клоны файлов (Clones) и мгновенные снимки (Snapshots). Чем отличается от традиционных ФС.
- [x] **Безопасность SSD**: Почему `shred` бесполезен на SSD и как работает Secure Erase (`blkdiscard` / NVMe Format).
- [x] **Cloud-native storage**: Signed URLs и Lifecycle Policies в S3. Особенности S3-совместимых хранилищ (MinIO).
- [x] **Git Internals**: Дополнить главу 47 схемой того, как Git хранит объекты (blobs, trees, commits).
- [x] **Переписать введение Части V (ch5/00-intro.md)**: Текущее введение — самое слабое из всех частей. Нужно: исправить заголовок («часть» вместо «глава»), добавить таблицу глав с ссылками (17 глав!), Mermaid-диаграмму архитектурных слоёв, блок `!!! abstract`.
- [x] **FUSE — файловые системы в user-space**: Добавить раздел про FUSE (sshfs, rclone mount, s3fs). Как создать простейшую ФС на Python (`fusepy`).
- [x] **Сравнительная таблица файловых систем**: ext4 vs XFS vs Btrfs vs ZFS vs APFS vs NTFS — CoW, снапшоты, макс. размер, журналирование, компрессия, дедупликация.
- [x] **tmpfs и ramfs**: Файловые системы в оперативной памяти — зачем нужны, чем отличаются друг от друга, типичные точки монтирования (`/dev/shm`, `/tmp`, `/run`). Применение: ускорение сборок, хранение секретов без записи на диск, shared memory между процессами. Аналоги на macOS (RAM-диск через `diskutil`).
